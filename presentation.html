3. Database Schema & Key Decisions

The database relies on SQLite for zero-configuration, local persistence. I maintained a strict separation of concerns:

  • raw_production_plan & raw_actual_production: Exact, untouched replicas of the CSVs. (Decision: Always keep raw data for auditability.)
  • production_plan & actual_production: Cleaned tables where column names are normalized and invalid rows are dropped.
  • exceptions: The materialized table storing computed deficits and operational statuses (open, acknowledged, resolved).

Key Decision (Idempotency): The ingestion and detection scripts were designed to be fully idempotent. The exceptions table uses a unique constraint on (product_code, date). This prevents data duplication and protects manual status updates if the pipeline is re-run.

4. Process, Data Findings, and Tradeoffs

Data Findings & Cleaning

  • Duplicates: Found 13 exact duplicate rows in the plan CSV. I dropped them to prevent skewed targets.
  • Invalid Targets: Found rows with zero/negative planned units and nulls. These were discarded since a deficit cannot be calculated against a non-positive target.
  • Mismatched Date Ranges: The actual production data only ran through Q1 (2017), whereas the plan data ran for a full year. I chose to skip plan rows without matching actuals rather than treating them as "0 produced." Treating them as 0 would have flooded the inbox with 9 months of false-positive exceptions.

Tradeoffs

  • No Heavy ORM Relationships: I kept the SQLAlchemy models flat to adhere strictly to the "no over-abstraction" rule. Direct queries were fast and sufficient for this scale.
  • Vanilla CSS over UI Libraries: Instead of heavy dependencies like Tailwind or Material UI, I used plain CSS to deliver a fast, clean interface focused on functionality.

5. What I Would Do Differently (Next Steps)

If I had more time to expand this project, I would implement the following:

  • Data Visualization: Replace the HTML table for the 7-day trend with a lightweight charting library (like Recharts) for quicker visual insights.
  • Pagination: Implement server-side pagination or infinite scroll on the /exceptions endpoint to handle larger datasets gracefully.
  • Authentication & Logging: Add basic JWT authentication to track which operator acknowledged or resolved an exception.
  • Automated Testing: Build out a pytest suite for the pipeline logic to guarantee edge cases (like the Q1 date cutoff) remain stable during refactors.
